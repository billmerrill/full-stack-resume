Introductory Letter

My name is Bill Merrill, I am web developer with comprehensive experience delivering internet products. I am applying for your Senior Front-End Web Developer position and would appreciate your consideration.

The right amount of process and communication feed back into themselves and make successes and problems evident.

Lastly, the LC III was my first Macintosh computer and I have used them as my primary computer ever since. I’ve used several of the Omni Group’s products over the years and I’d be very pleased to support their sale and use.

Example Work

Topophile

website

Topophile is a hobby business I’ve created to explore geographical data and geo-programming. The website gives a person familiar with online tools like MapQuest or Google Maps tools to 3D print custom models of the Earth’s surface. I am responsible for both the front and backend of the website and I’ll take this opportunity to talk about a couple points in the front-end.

The landing page, the first thing most users will see, was the last thing I completed before the initial launch. While the primary app did not work on mobile in 2014 (I’m now excited to see WebGL has become standard on iOS Safari and I look forward to improving the model builder for mobile) I did make sure the page was responsive and looked good on smart phone browsers.

[ INSERT Responsive Landing Page Image] [ Caption: Showing the Topophile landing page in desktop and smartphone views ]

The model builder has three steps, the first is exploring terrain. Rather than attempt a VR-like experience, I strove to use a familiar tiled web map and show the terrain related to selected areas. On the left a person searches for a feature and selects an area on the map, on the right a low-resolution model of the terrain is quickly displayed. I wanted clarity and speed for this step as users spend some time perfecting their selection.

[ INSERT Terrain search Image] [Caption: Animation, the process of searching and refining terrain for a model]

I found communicating the size of printed model to be very important as people decided to purchase. I started by selecting three standard sizes for a model and displayed the height, width, and depth of a model textually. Going further I wanted to show an object a person might have handy for comparison. I chose a US dollar bill and a Euro note and added them to the 3D scene. I hoped that seeing a object the could hold in real life might help them image the size of a plastic printed model.

In testing, I found people were reluctant to click the buttons. They thought the buttons were related to the purchase price rather than the size of the model. The next time I work on this element I will seek better solutions.

[INSERT Size Compare Image] [Caption: Animation, comparing different size models with real world objects]

I made a simple and I think beautiful clock animation using CSS to show the steps of unknowable duration that lead from a person designing a model to being able to purchase it from my 3D printing vendor. While functionally no different than a spinner, I liked the visual representation of the steps setting up expectations and letting you know something’s still happening.

I’ve composed this application as a HTML5 document.

I starting writing prose in Pages to take advantage of the standard writing tools like spelling and grammar checking. I made notes of images and animations I’d like to include and put placeholders in the prose.

Then I created a source control repository for the project and added a blank html file and a directory for media assets. I edited the html file using the Atom code editor and filled out a skeleton of tags with the basic structure of 3 sections, the Introduction, the Résumé, and the Example works. Later I pasted sections of prose into the skeleton, linked images and animations, and made simple style adjustments.

I used the built-in macOS tools to create screens shots of my websites and record videos for animations. I edited my screen shots using Preview.app and Acorn; I used the command line tool ‘ffmpeg’ to convert the videos to animated gifs.

During the creation of this document I hosted all the assets on a local apache2 httpd instance. I enjoy maintaining local development environments with few external dependencies, a habit from days of less-than-reliable network connectivity.

As this is a simple document I’m not too worried about cross-browser display issues, but given the audience I’ve checked it in Omniweb, Safari, Chrome, Firefox on my Macbook Pro and Safari on my iPhone and iPad.

I’ve also checked the webpage using an HTML5 validator. I’ve not found it is economical to chase all the issues a validator might present, but they do often point out deficiencies that should be addressed such as illegally used tags or missing attributes. Standards compliance goes a long way toward making an accessible document. When working with Javascript I find jslint to be invaluable in a similar capacity.
